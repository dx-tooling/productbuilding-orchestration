# Plan: Natural Conversational Notifications

## Goal

Replace synthetic template-filled Slack messages (generated by the notifier when GitHub webhooks arrive) with LLM-backed conversational responses. All communication visible to the user must flow through the agent so the bot has a consistent, natural voice. Additionally, fix two related agent self-awareness bugs exposed during analysis.

## Guiding principles

- **Red-green TDD throughout.** Every behaviour change is preceded by a failing test. The test is committed before the implementation. No production code changes without a red test first.
- **Phase transitions are non-negotiable.** `updatePhaseForEvent` in the notifier must run unconditionally regardless of whether a Slack message is posted. These DB state changes drive the router.
- **Thread creation is non-negotiable.** The notifier's `PostMessage` + `SaveThread` path for new threads must remain intact. Only reply messages within existing threads are being replaced.
- **Blast radius is minimal.** Each step is independently releasable. No step requires the next to be complete before it is safe to deploy.

## Background: three distinct bugs

### Bug 1 — Webhook echo (notifier problem)

The bot closes an issue. GitHub fires a webhook. The notifier receives it within milliseconds and posts `"This issue has been closed."` to the thread before the agent's own response is generated. The user sees a robotic template message followed by a confused agent reply.

**Fix**: Parse `sender.login` from GitHub webhook payloads. Compare against the configured bot GitHub login (`BotGitHubLogin` in `TargetConfig`). When `sender == bot`, the event is bot-initiated — suppress the notifier's template reply (but keep phase transitions). This is the same sender-detection mechanism used to decide `shouldDelegateToAgent` for external events.

### Bug 2 — Confused agent response (LLM quality problem)

After calling `close_github_issue` and receiving `"Closed issue #95."` in its tool result, the agent responds with "It looks like #95 is actually already closed!" — language of discovery, not intention. The closer prompt permits the LLM to reason about observed state rather than asserting its own action. `SideEffects` also lacks tracking for closed issues/PRs (asymmetric with `CreatedIssues`).

**Fix**: (a) Tighten the closer prompt. (b) Add `ClosedIssues`/`ClosedPRs` to `SideEffects` for programmatic tracking.

### Bug 3 — Bot comments echoed back to Slack (incomplete loop prevention)

After the agent delegates work to OpenCode (`"I'll keep you posted as progress comes in."`), the following three synthetic messages appear in the Slack thread:

```
@opencode-agent[bot] commented on GitHub: Created PR #96         ← EventCommentAdded
@opencode-agent[bot] opened a pull request, touching 51 lines.  ← EventPROpened
@PrdctBldr commented on GitHub: Preview deploying...            ← EventCommentAdded
```

The third message is the bot narrating its own preview-deployment checklist comment back to Slack. The existing `<!-- via-agent -->` / `<!-- via-slack -->` content filter does not catch it because the preview service posts that comment directly without those markers.

**Fix**: Extend `handleIssueComment` to also skip comments where `sender.login == BotGitHubLogin` (sender-based filter, complementing the existing content-based filter). This suppresses the preview checklist echo at the webhook handler level before it reaches the notifier.

The first two messages (from `opencode-agent[bot]`) are legitimate external-actor events that carry useful information; they should be routed to the agent invoker for natural-language translation rather than suppressed.

## What gets removed vs. replaced

### Safe to remove (bot self-narration, no replacement needed)

| Event | Current message | Why safe |
|---|---|---|
| `EventIssueOpened` reply | `"@author opened this issue."` | Agent already replied when it created the issue; thread parent message still anchors the thread |
| `EventIssueReopened` | `"This issue has been reopened."` | Usually bot-initiated; no phase transition |
| `EventIssueClosed` | `"This issue has been closed."` | Usually bot-initiated; no phase transition |
| `EventPRClosed` | `"This pull request has been closed…"` | Bot-initiated; no phase transition |
| `EventCIPassed` | `"CI checks are passing."` | Passive non-event; no user action required |
| `EventPROpened` in `PhaseOpen` | `"Work has started on your request."` | Agent already replied when it delegated; `PhaseOpen→PhaseInProgress` transition must still run |

### Must be replaced with LLM agent invocation

| Event | Why critical |
|---|---|
| `EventPRReady` | Delivers preview URL; triggers `PhaseReview` transition + `SetPreviewNotified`; user is waiting |
| `EventPRFailed` | User is waiting for preview; agent should explain and offer to investigate |
| `EventCIFailed` | Time-sensitive; agent should proactively report and offer to investigate |
| `EventPRMerged` | "This is live" confirmation; `PhaseDone` transition |
| `EventCommentAdded` from external actors | Must remain in thread history; agent should translate to natural language. "External actor" includes both human users and automation bots like `opencode-agent[bot]`. |
| `EventPROpened` from external actors (non-PhaseOpen) | PR opened by OpenCode or a human outside the normal PhaseOpen flow; agent should confirm and contextualise (e.g. "OpenCode has opened PR #96 — preview is deploying"). |

### Suppress without replacement (bot self-narration via content-agnostic sender check)

| Event | Why suppress |
|---|---|
| `EventCommentAdded` where `sender == BotGitHubLogin` | Bot's own GitHub comments (preview checklist, status updates) echoed back to Slack. The existing `<!-- via-agent -->` / `<!-- via-slack -->` content filter does not catch preview-service comments — sender check is required. Suppressed at webhook handler level, before the notifier. |

#### Green

Add to `notifier.go`:

```go
// shouldSkipMessage returns true for events where message generation
// is either unnecessary (bot self-narration) or delegated to the agent invoker.
func shouldSkipMessage(eventType slackfacade.EventType, isBotInitiated bool) bool {
    switch eventType {
    case slackfacade.EventIssueOpened:    return true // reply only; parent message is still posted
    case slackfacade.EventIssueReopened:  return true
    case slackfacade.EventCIPassed:       return true
    case slackfacade.EventPRReady:        return true // agent invoker handles this
    case slackfacade.EventPRFailed:       return true // agent invoker handles this
    case slackfacade.EventCIFailed:       return true // agent invoker handles this
    case slackfacade.EventPRMerged:       return true // agent invoker handles this
    case slackfacade.EventCommentAdded:
        // Bot's own comments are suppressed at webhook-handler level (Step 6) before
        // reaching the notifier, so by the time we get here the sender is always an
        // external actor. Keep the relay so the comment appears in thread history for
        // the agent's ThreadMessages context. The agent invoker fires separately.
        return false
    }
    if isBotInitiated {
        return true // suppress all other bot-echo events
    }
    return false
}
```

In `flush()`, gate the `PostToThread` call:

```go
if !newThread && !shouldSkipMessage(event.Type, isBotInitiated) {
    updateMsg := n.messages.EventMessage(...)
    // post...
}
```

The `isBotInitiated` flag is derived from `event.Author == target.BotGitHubLogin` (passed through `NotificationEvent`) or from the sender field stored on the event.

**Why `EventCommentAdded` still reaches the notifier**: bot-authored comments are blocked at the webhook handler level in `handleIssueComment` (Step 6) — they return early before calling the notifier. Comments that do reach the notifier are therefore always from external actors (humans or `opencode-agent[bot]`). These should remain visible in the Slack thread (thread history) so the agent's `ThreadMessages` context includes them on the next invocation.

## Risk register

| Risk | Mitigation |
|---|---|
| Phase transitions silently skipped | `updatePhaseForEvent` is unconditional — gating is only on `PostToThread`. Tests in Step 9 verify phase transitions independently of message posting. |
| Race: agent invoker looks up thread before it is created | `EventAgentInvoker` retries once after `retryWait` (default 5s, 10ms in tests). Tested in Step 7. |
| Agent comment loop via GitHub → Slack → GitHub | Existing `<!-- via-agent -->` tagging prevents the agent's own Slack-driven comments re-entering. The new sender-based filter (Step 6) prevents preview-service comments re-entering. Both checks must remain. |
| LLM latency in webhook response path | All agent invocations are in goroutines (`go h.agentInvoker.InvokeForEvent(...)`). Webhook returns 202 immediately. |
| `EventPRReady` `SetPreviewNotified` / `SetFeedbackRelayed` skipped | These are in `updatePhaseForEvent`, which is unconditional. Step 9 tests assert these flags are still set. |
| `BotGitHubLogin` not configured in existing deployments | Default to empty string; when empty, sender check is skipped and all events are treated as potentially external. Safe fallback. |
| `EventIssueOpened` parent message removed | Only the reply `"@author opened this issue."` is removed. `PostMessage` (thread creation) and `SaveThread` continue unconditionally — they are not gated by `shouldSkipMessage`. |
| Two close-together events from `opencode-agent[bot]` generate two agent responses | The comment "Created PR #96" and the `EventPROpened` webhook fire within milliseconds. Each independently invokes the agent, potentially producing two Slack messages. Known limitation — acceptable for now. Future improvement: debounce agent invocations per thread with a short window, same as the notifier's existing debouncer. |
